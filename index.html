Training Material

Interacting with Engineers

I built this training deck for non-technical program staff who needed to write specifications for software developers. The core insight: if a developer understands clearly what you're trying to do, they'll often have a better idea of how to do it. A PM's job is to make that understanding possible.

  • Separate business logic from implementation details Specs should answer WHAT and WHY in plain language before addressing HOW in technical terms. When these are tangled together, developers can't apply their own expertise to the solution.
  • User stories anchor every feature "As a ___, I want ___, so that ___." One sentence. Zero implementation details. This clarifies WHO the feature serves and WHY it matters before a single line of code is discussed.
  • Define complex concepts once, reuse by name If a condition or workflow appears repeatedly in a spec, define it once with both its business meaning and its technical translation. Then refer to it by name. This eliminates copy-paste drift and keeps specs maintainable.

This is the same skill as defining platform contracts: making a system's capabilities legible to the teams building on it, so they can build effectively without needing deep infrastructure expertise.

Case Study

Measuring What Matters

During the COVID-19 pandemic, I supported a state public health agency that needed to rapidly scale its contact tracing operations. The critical workflow was bulk case assignment: supervisors had to assign batches of newly reported cases to individual caseworkers every morning. If this process was slow, cases sat unassigned and contacts went untraced.

1
Identified the problem The bulk assignment form was slow enough to be a daily bottleneck. But nobody had measured where the time was actually going.
2
Measured it manually I sat down with a stopwatch and timed each step: form load time, case list rendering, assignment submission, confirmation screen. Raw numbers, no guessing.
3
Set thresholds collaboratively I worked with the division lead to establish acceptable performance targets for each step. These weren't arbitrary benchmarks; they were grounded in the operational reality of morning case assignment.
4
Built it into the system Through this process, I identified the key workflows to instrument permanently. The manual stopwatch exercise became the blueprint for automated performance monitoring, so nobody had to do that again.

I care about metrics because I've seen what happens without them. The goal isn't dashboards for the sake of dashboards; it's making performance visible so that teams can act on it before users feel the pain. This approach — measure manually, set meaningful thresholds, then bake the instrumentation into the system — is how I think about platform observability.

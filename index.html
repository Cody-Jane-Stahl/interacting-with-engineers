Training Material

Interacting with Engineers

I built this training deck for non-technical program staff who needed to write specifications for software developers. The core insight: if a developer understands clearly what you're trying to do, they'll often have a better idea of how to do it. That's the PM's job — close that gap.

1
Separate business logic from implementation details Specs should answer WHAT and WHY in plain language before addressing HOW in technical terms. When these are tangled together, developers can't apply their own expertise to the solution.
2
User stories anchor every feature "As a ___, I want ___, so that ___." One sentence. Zero implementation details. This clarifies WHO the feature serves and WHY it matters before a single line of code is discussed.
3
Define complex concepts once, reuse by name If a condition or workflow appears repeatedly in a spec, define it once with both its business meaning and its technical translation. Then refer to it by name. Otherwise the same condition ends up written six different ways across the spec, and three of them are wrong.

I built this because specs were coming in broken in every direction — some were pure implementation details with no business context, others were so vague that developers had to guess, and we kept building the wrong thing. The same skill applies to platform work: if you can understand what domain teams need in their own language, you can translate it to platform language and back. That's how you build a search platform that teams actually use instead of work around.

Case Study

Measuring What Matters

During the COVID-19 pandemic, I supported a state public health agency that was standing up contact tracing operations fast. The critical workflow was bulk case assignment — supervisors assigned batches of new cases to caseworkers every morning. If that process was slow, cases piled up and contacts went untraced.

1
Identified the problem The bulk assignment form was slow enough to be a daily bottleneck. But nobody had measured where the time was actually going.
2
Measured it manually I sat down with a stopwatch and timed each step: form load time, case list rendering, assignment submission, confirmation screen. Raw numbers, no guessing.
3
Set thresholds collaboratively I worked with the division lead to establish acceptable performance targets for each step. These came from the actual workflow — how many cases hit the queue overnight, how fast supervisors needed to clear them before the morning standup.
4
Built it into the system Through this process, I identified the key workflows to instrument permanently. That stopwatch session told us exactly which workflows to instrument, so nobody had to sit there timing forms by hand again.

I care about metrics because I've been in the room when nobody has them. You end up arguing about whether something is slow based on vibes. The stopwatch was step one; the point was to turn that into permanent instrumentation so the data is just there next time someone asks.
